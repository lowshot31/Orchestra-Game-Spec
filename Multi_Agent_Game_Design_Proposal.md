# 기술 제안서: Orchestra (Tutti) - 실무 도입을 위한 멀티 에이전트 파이프라인

## 1. Executive Summary: 베이글코드의 고민에 대한 해답

베이글코드 모바일 캐주얼팀의 과제 핵심은 명확합니다. 
> *"기획자는 에이전트에게 게임을 만들게 하고, 엔지니어는 에이전트의 능력을 확장합니다."*

**Orchestra(Tutti)**는 단순한 장난감 챗봇이 아닙니다. 이 도구는 철저하게 **"비개발 직군의 사용성"**과 **"엔지니어의 확장성"**이라는 두 마리 토끼를 잡기 위해 설계되었습니다.

1. **기획자를 위한 UX (Discord & HitL):** 터미널을 열 필요 없이, 익숙한 Discord 환경에서 `/tutti` 커맨드 하나로 4명의 에이전트를 오케스트레이션합니다. 
2. **엔지니어를 위한 확장성 (Adapter & Decoupling):** 특정 LLM에 종속되지 않습니다. Provider 패턴을 적용하여, 엔지니어는 기존 코드 수정 없이 새로운 에이전트나 모델(Claude, Gemini, Local Ollama)을 무한히 확장(Pluggable)할 수 있습니다.

## 2. 과제 요구사항 완벽 대응 (Compliance Matrix)

| 베이글코드 요구사항 | Orchestra (Tutti)의 해결책 |
| :--- | :--- |
| **두 개 이상의 에이전트 통신** | 🎨 Designer, 🛠️ Reviewer, 💼 CEO, ✍️ Writer 총 4개의 에이전트가 역할을 분담하여 메시지를 주고받으며 파이프라인을 형성합니다. |
| **사용자 관찰 및 개입 (HitL)** | 에이전트 간의 대화는 Discord 쓰레드에 실시간 중계되며, CEO 에이전트의 Risk 평가에 따라 파이프라인이 멈추고 사용자의 **수정/승인 개입**을 기다립니다. |
| **통신 방식 / UI 자유** | 비개발자도 접근 가능한 **Discord 봇 UI**와 엔지니어를 위한 **CLI 환경**을 동시에 제공하며, 독립적인 Workflow Engine으로 통신을 제어합니다. |
| **AI 코딩 에이전트 활용** | 초기 아키텍처부터 최종 문서화까지 AI 에이전트(Codex, Gemini CLI 등)와 페어 프로그래밍으로 E2E 구현되었습니다. |
| **README 기반 즉시 동작** | 외부 API 키 세팅 없이도 즉시 실행 가능한 `Mock` 모드를 기본 제공하여, 평가자가 `clone` 후 단 1분 만에 동작을 검증할 수 있습니다. |

## 3. 핵심 아키텍처 의도 (Why this design?)

### 3.1. 에이전트 역할의 수직 분리 (Separation of Concerns)
하나의 거대한 프롬프트로 모든 것을 처리하지 않습니다. 
- **Designer**는 창의적인 확산(Divergence)을, **Reviewer**는 기술적 수렴(Convergence)을 담당합니다. 
- **CEO**는 산출물의 Risk를 평가하여 자율 주행을 할지, 인간(Human)에게 운전대를 넘길지 판단합니다. 
- 이 구조는 향후 QA 에이전트, 사운드 에이전트 등을 쉽게 끼워 넣을 수 있는 파이프라인 기반을 제공합니다.

### 3.2. 인간 개입의 최적화 (Risk-based Human-in-the-Loop)
모든 과정에 인간이 개입하면 자동화의 의미가 퇴색됩니다. Tutti는 **선택적 개입**을 구현했습니다.
- `Risk: LOW` -> 즉시 자동 승인 (시간 단축)
- `Risk: MEDIUM` -> 15초 대기 후 자동 진행 (관찰 및 옵션 개입)
- `Risk: HIGH` -> 파이프라인 정지 후 인간의 결재 대기 (안전망)

### 3.3. 프로토타입 및 GitHub PR 자동화 (Zero to PR)
AI의 결과물이 채팅창에 머무는 것은 실무적이지 않습니다.
Tutti는 최종 기획 명세가 떨어짐과 동시에, **브라우저에서 즉시 플레이 가능한 HTML/JS 게임**을 생성하고, 개발팀 핸드오프를 위해 **GitHub PR까지 자동으로 생성**합니다. 외부 라이브러리(`requests` 등) 없이 순수 Python 표준 라이브러리로만 구현하여 의존성을 최소화했습니다.

## 4. 엔지니어링 확장성 (How Engineers Extend it)

엔지니어는 파이프라인 내부 로직을 건드리지 않고도 시스템을 확장할 수 있습니다.

1. **Model Adapter 패턴:** `adapters.py`에 새로운 클래스만 상속받아 추가하면 어떤 LLM이든 파이프라인에 주입 가능.
2. **지식 베이스 주입:** `knowledge.py`를 통해 프로젝트별(예: 두더지잡기, 슬롯머신) 특화 규칙을 에이전트 프롬프트에 동적(Dynamic)으로 주입.
3. **Stateless Workflow:** 상태 관리가 추상화되어 있어, 향후 데이터베이스(SQLite, Redis)를 붙여 영구적인 세션 관리를 도입하기 용이한 구조.

## 5. 결론: "실험"을 넘어 "실무"로

Tutti는 "AI가 대화하는 신기한 데모"를 넘어, 실제 프로덕션 환경에서 **팀 단위로 사용될 수 있는 구조**를 고민한 결과물입니다. 

기획자는 Discord에서 아이디어만 던지면 완성된 기획서와 플레이어블 빌드를 얻고, 엔지니어는 GitHub PR을 통해 코드 뼈대를 전달받아 비즈니스 로직에만 집중할 수 있습니다. 

Orchestra(Tutti)는 베이글코드가 지향하는 **AI Native 제품 개발 파이프라인**의 강력한 프로토타입이 될 것입니다.
