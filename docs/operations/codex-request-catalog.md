# Codex 요청문 모음

## 문서 목적

이 문서는 창업자가 Codex에게 반복적으로 요청할 수 있는 표준 문장을 정리한 문서다.

목표는 아래와 같다.

1. 매번 새로 설명하느라 생기는 낭비를 줄인다
2. `GPT-5.5 메인 운영자 + GPT-5.4 서브 에이전트` 역할 분리를 유지한다
3. gstack 스킬을 언제 어떻게 태울지 일관되게 만든다

이 문서의 요청문은 그대로 복붙해서 써도 되고, 현재 작업에 맞게 한두 줄만 바꿔서 써도 된다.

---

## 기본 사용 규칙

### 1. 요청은 모드를 먼저 정하고 시작한다

가능한 한 아래 중 하나를 먼저 고른다.

- `planning mode`
- `sprint mode`
- `triage mode`
- `release mode`
- `maintenance mode`

### 2. 메인 모델과 서브 에이전트 역할을 분리한다

기본 문구:

```text
너는 GPT-5.5 메인 운영자처럼 행동해라.
GPT-5.4 서브 에이전트는 범위가 좁은 조사, 구현, 테스트 작업에만 사용해라.
제품 방향, 범위 판단, 아키텍처 결정은 메인 모델이 맡아라.
```

### 3. 요청마다 반드시 넣으면 좋은 정보

- 문제 또는 목표
- 현재 상태
- 원하는 종료 상태
- 범위 제한
- 지금 하지 않을 것

## 11. 짧은 한 줄 요청문

빠르게 던질 때 쓰는 짧은 버전이다.

### 기획

- `이거 planning mode로 정리해줘. 아직 구현하지 말고 범위만 잘라줘.`
- `이 아이디어가 이번 주 기능인지, 에픽인지, 보류감인지 판단해줘.`

### 구현

- `이 기능 sprint mode로 돌려줘. 메인은 판단, 서브는 좁은 구현만 맡겨줘.`
- `이걸 작업자 단위로 쪼개줘. 서로 파일 범위 안 겹치게.`

### 리뷰

- `이 변경 review mode로 검토해줘. 운영에서 터질 것만 우선 잡아줘.`
- `이건 보안 민감하니까 cso 관점까지 같이 봐줘.`

### QA

- `이거 qa mode로 검증해줘. 운영자 입장에서 불편한 지점까지 잡아줘.`
- `수정 없이 qa-only 리포트만 줘.`

### 배포

- `이거 release mode로 ship/no-ship 판단해줘.`
- `배포 전 빠진 체크리스트가 있는지 봐줘.`

### 유지보수

- `이 영역 maintenance mode로 정리해줘. 안정성과 운영 단순화 중심으로.`
- `이거 당장 고칠 것과 나중에 에픽으로 뺄 것을 나눠줘.`

## 12. 요청문 작성 규칙

좋은 요청문은 보통 아래 구조를 따른다.

```text
1. 어떤 모드인지
2. 메인/서브 역할 규칙
3. 목표
4. 현재 상태
5. 범위 제한
6. 원하는 출력 형태
```

예시:

```text
이번 작업을 sprint mode로 진행해줘.
너는 GPT-5.5 메인 운영자다.
GPT-5.4 작업자는 좁은 구현과 테스트만 맡겨라.
목표는 승인 큐 MVP를 리뷰 준비 상태까지 끌고 가는 것이다.
이번 범위에는 대시보드 전체 개편은 넣지 마라.
출력은 작업 분해 -> 구현 계획 -> 실제 진행 순서로 정리해라.
```
