# 창업자 + AI 팀 운영 가이드

## 문서 목적

이 문서는 다음 조합으로 1인 창업 소프트웨어 회사를 운영하는 방법을 정의한다.

- 메인 모델: `GPT-5.5`
- 서브 에이전트: `GPT-5.4`
- 워크플로 레이어: `gstack` 스킬

현재 1차 대상은 `내부업무/에이전트 툴을 만드는 1인 창업자`다. 다만 구조 자체는 아래 순서로 확장 가능하도록 설계한다.

1. 창업자 1인 + AI 팀
2. 소규모 인간 팀 + AI 팀
3. 정식 소프트웨어 회사 운영 체계

이 문서는 아래 질문에 답할 수 있어야 한다.

- 일은 어떤 방식으로 들어오는가
- 제품 결정과 회사 운영 결정은 어떻게 내려지는가
- 메인 모델은 언제 판단을 맡는가
- 서브 에이전트는 언제 실행을 맡는가
- 각 단계에서 어떤 gstack 스킬을 쓰는가

## 운영 원칙

### 1. 크게 계획하고 작게 실행한다

회사는 `에픽/프로젝트` 단위로 생각하고, 실행은 `기능/작업` 단위로 쪼갠다.

- 큰 목표는 에픽으로 관리한다
- 에픽은 배포 가능한 기능으로 나눈다
- 기능은 다시 좁고 명확한 작업으로 나눈다
- 작업은 한 명의 작업자 또는 한 개의 서브 에이전트가 처리할 수 있을 정도로 작아야 한다

### 2. 메인 모델은 판단하고, 서브 에이전트는 실행한다

역할을 강하게 분리해야 토큰 낭비를 줄이고 결정의 일관성을 유지할 수 있다.

`GPT-5.5 메인 모델`

- 제품 문제 정의
- 우선순위 결정
- 회의 진행과 정리
- 아키텍처 판단
- 승인 게이트 관리
- 최종 리뷰와 출시 여부 판단

`GPT-5.4 서브 에이전트`

- 코드베이스 조사
- 범위가 좁은 구현
- 테스트 작성
- 문서 보강
- QA 재현
- 제한된 범위의 리팩터링

서브 에이전트는 제품 방향을 결정하지 않는다. 서브 에이전트는 확정된 결정을 실행하는 역할을 맡는다.

### 3. 사람의 승인은 레버리지 지점에서만 받는다

창업자가 모든 작은 행동을 승인할 필요는 없다. 대신 잘못 선택했을 때 범위, 비용, 리스크가 크게 바뀌는 지점에서만 승인한다.

필수 승인 지점:

- 어떤 에픽을 지금 진행할지 정할 때
- 기능 범위를 최종 확정할 때
- 중간 이상 리스크가 있는 설계를 승인할 때
- 보안, 안정성, 속도 간 트레이드오프를 수용할 때
- 실제 배포 여부를 판단할 때

### 4. 처음에는 바텀업 방식으로 회사를 만든다

지금 단계에서는 큰 조직 흉내를 낼 필요가 없다. 대신 아래를 지키는 것이 중요하다.

- 가볍지만 반복 가능한 주간 운영 리듬
- 짧고 오래가는 문서
- 잦은 배포
- 배포 후 학습 내용 축적

### 5. 모든 배포물은 추적 가능해야 한다

1인 회사라도 의미 있는 변경은 모두 아래와 연결되어야 한다.

- 어떤 문제였는지
- 어떤 결정을 내렸는지
- 어떻게 구현했는지
- 어떤 리뷰를 거쳤는지
- 어떻게 배포했는지
- 무엇을 배웠는지

## 팀 구조

### 핵심 역할

`창업자`

- 회사 방향을 정한다
- 사업 우선순위를 정한다
- 범위와 배포 여부를 승인한다

`GPT-5.5 메인 운영자`

- PM, Chief of Staff, 아키텍트, 릴리즈 코디네이터 역할을 겸한다
- 전체 워크플로를 운영한다
- 적절한 gstack 스킬을 고른다
- 필요 시 서브 에이전트에 위임한다
- 여러 결과를 하나의 의사결정 흐름으로 정리한다

`GPT-5.4 작업자`

- 좁은 범위의 전문 실행자 역할을 맡는다
- 조사, 구현, 테스트, 문서 보강 같은 작업을 병렬로 수행한다
- 결과물이나 패치를 반환한다

### 기본 작업 규칙

여러 도메인의 판단이 필요한 일은 메인 모델이 맡는다.

범위가 좁고, 병렬화 가능하고, 모호성이 낮은 일은 5.4 작업자에게 위임한다.

## 일의 계층 구조

아래 구조를 일관되게 사용한다.

### 1. 회사 테마

예시:

- 내부 운영 속도를 높인다
- 수작업 지원 업무를 줄인다
- 에이전트 신뢰성을 높인다

### 2. 에픽

예시:

- 관리자 워크플로 자동화 구축
- 승인 큐 시스템 추가
- 에이전트 관측성 개선

### 3. 기능

예시:

- 감사 로그 패널
- 재시도 큐 화면
- 작업자 상태 요약 뷰

### 4. 작업

예시:

- 백엔드 API 엔드포인트 추가
- 테이블 컴포넌트 구현
- 재시도 큐 테스트 작성

## 회사 운영 흐름

이 흐름은 회사 단위의 주간 운영 루프다.

### 주간 운영 루프

1. 문제, 아이디어, 반복되는 불편을 수집한다
2. 현재 진행 중인 에픽을 검토한다
3. 이번 주에 중요한 것을 정한다
4. 우선순위를 기능 단위로 변환한다
5. 제품 개발 파이프라인을 실행한다
6. 배포한다
7. 결과를 리뷰하고 학습 내용을 남긴다

### 주간 창업자 회의

이 회의는 메인 모델과 함께 주 1회 진행한다.

기본 아젠다:

1. 지난주에 실제로 무엇이 바뀌었는가
2. 지금 막혀 있는 것은 무엇인가
3. 당장 중요한 것은 무엇인가
4. 가장 큰 레버리지를 만드는 것은 무엇인가
5. 이번 주에 과감히 잘라야 할 것은 무엇인가
6. 이번 주에 반드시 배포할 것은 무엇인가

추천 gstack 스킬:

- `gstack-office-hours`: 문제 재정의, 기회 탐색
- `gstack-plan-ceo-review`: 범위와 전략 압박 테스트
- `gstack-retro`: 지난 사이클 리뷰

### 일일 운영 체크

매 작업 세션 시작 전에 짧게 확인한다.

질문:

1. 오늘 가장 중요한 기능 또는 이슈는 무엇인가
2. 이미 진행 중인 것은 무엇인가
3. 무엇이 막혀 있고 누가 해결해야 하는가
4. 오늘 필요한 모드는 기획, 구현, 리뷰, QA, 배포, 유지보수 중 무엇인가

메인 모델은 아래 중 하나의 실행 모드를 선택해 답해야 한다.

- `Planning mode`
- `Build mode`
- `Review mode`
- `QA mode`
- `Release mode`
- `Maintenance mode`

## 제품 개발 흐름

이 흐름은 대부분의 기능 개발에 기본으로 적용한다.

### 1단계. 일감 유입

입력 소스:

- 창업자 아이디어
- 운영 중 발견한 불편
- 버그 리포트
- 지원 과정에서 드러난 마찰
- 안정성 문제

출력:

- 문제 정의 1개
- 원하는 결과 1개
- 초기 범위 가설 1개

### 2단계. 제품 프레이밍

목표:

모호한 아이디어를 실제 문제와 첫 번째 유효한 범위로 정리한다.

사용 스킬:

- `gstack-office-hours`
- `gstack-plan-ceo-review`

메인 모델 책임:

- 진짜 사용자/운영자 고통이 무엇인지 찾는다
- 불필요한 범위를 잘라낸다
- 가장 작은 유효 버전을 정의한다

출력:

- 에픽 또는 기능 요약
- 대상 사용자
- 성공 기준
- 비목표

### 3단계. 설계와 기술 계획

목표:

구현이 여러 파일로 퍼지기 전에 기능의 동작 방식을 확정한다.

사용 스킬:

- `gstack-plan-eng-review`
- `gstack-plan-design-review`: UI나 운영 화면 경험이 중요한 경우
- `gstack-plan-devex-review`: 사용자가 개발자나 운영자인 경우

메인 모델 책임:

- 아키텍처 정의
- 경계와 인터페이스 정의
- 승인 기준 정의
- 테스트 전략 정의

출력:

- 승인된 설계
- 명확한 작업 분해
- 명시된 리스크

### 4단계. 작업 분해

승인된 기능을 아래처럼 나눈다.

- 메인 조정 작업 1개
- 구현 작업 1~4개
- QA/테스트/문서 작업 0~2개

위임 규칙:

- 각 작업자는 겹치지 않는 소유 영역을 가져야 한다
- 두 작업자에게 같은 파일 범위를 주지 않는다
- 아직 확정되지 않은 제품 결정을 서브 에이전트에 넘기지 않는다

### 5단계. 구현

메인 모델 역할:

- 작업 경계를 정한다
- 고판단도가 필요한 막힌 일을 직접 처리한다
- 작업자 결과물을 통합한다

5.4 작업자 역할:

- 특정 모듈 구현
- 테스트 추가
- 문서 업데이트
- 제한된 범위의 조사

좋은 작업자 위임 예시:

- retry queue API를 한 모듈 안에서 구현
- audit log 페이지의 테이블과 필터 UI 구현
- 특정 실패 시나리오에 대한 회귀 테스트 작성
- 현재 auth 흐름을 조사하고 edge case를 정리

좋지 않은 작업자 위임 예시:

- 전체 아키텍처를 설계해달라고 요청
- MVP 범위를 정해달라고 요청
- 넓은 코드베이스를 경계 없이 통째로 고치게 하는 것

### 6단계. 리뷰

사용 스킬:

- `gstack-review`
- `gstack-cso`: 보안 민감도가 높은 변경일 때

목표:

QA나 배포 전에 운영 환경에서 터질 문제를 미리 잡는다.

메인 모델 책임:

- 리뷰 결과를 정리한다
- 지금 반드시 고쳐야 할 항목을 결정한다
- 필요 시 좁은 수정 작업을 다시 작업자에게 위임한다

### 7단계. QA

사용 스킬:

- `gstack-qa`
- `gstack-qa-only`: 수정 없이 검증만 할 때
- `gstack-benchmark`: 성능 영향이 중요할 때

목표:

실제 운영자 시점에서 기능이 기대대로 동작하는지 확인한다.

출력:

- 통과 여부
- 버그 목록
- 필요한 회귀 테스트 항목

### 8단계. 배포

사용 스킬:

- `gstack-ship`
- `gstack-land-and-deploy`
- `gstack-canary`

목표:

리뷰와 QA를 통과했고 운영 비용을 감수할 가치가 있을 때만 배포한다.

배포 전 체크리스트:

- 범위가 승인된 기능과 일치하는가
- 핵심 테스트가 존재하는가
- 리뷰 결과가 해결되었거나 의도적으로 이월되었는가
- QA가 끝났는가
- 동작이 바뀌었다면 문서가 업데이트되었는가

### 9단계. 유지보수와 학습

사용 스킬:

- `gstack-document-release`
- `gstack-retro`
- `gstack-learn`

목표:

같은 비용을 반복해서 치르지 않도록 다음 작업을 더 쉽게 만든다.

남겨야 할 기록:

- 무엇이 깨졌는가
- 무엇이 예상보다 비쌌는가
- 무엇을 표준 패턴으로 삼아야 하는가

## 기본 모델 라우팅

### GPT-5.5 메인 모델

아래 작업은 메인 모델이 맡는다.

- 에픽 정의
- 회의 정리와 의사결정 요약
- 우선순위 조정
- 아키텍처 선택
- 리스크 트레이드오프 판단
- 배포 여부 결정
- 여러 작업자 결과 통합

### GPT-5.4 작업자

아래 작업은 작업자에게 맡긴다.

- 국소적인 코드 조사
- 좁은 범위의 구현
- 단위/통합 테스트 추가
- 문서 수정
- 범위가 좁은 버그 수정
- 반복 가능한 QA 작업

### 위임 규칙

아래 3개가 모두 참일 때만 위임한다.

1. 작업 범위가 명확하다
2. 바로 다음 핵심 의사결정이 아니다
3. 수정할 파일 범위를 구체적으로 말할 수 있다

하나라도 아니면 메인 모델이 맡는다.

## 표준 실행 모드

창업자는 가능한 한 적은 수의 고정된 요청 패턴으로 메인 모델을 움직이는 것이 좋다.

### 1. Planning Mode

문제가 아직 모호할 때 사용한다.

프롬프트:

```text
이건 우리 내부 툴 회사를 위한 기획 작업이다.
너는 GPT-5.5 메인 운영자처럼 행동해라.
적절하면 gstack 워크플로를 사용해라.
목표는 이 아이디어를 구현 전 단계의 승인된 기능 계획으로 바꾸는 것이다.

아이디어:
<문제 설명>
```

### 2. Sprint Mode

기능을 계획에서 구현 단계로 넘길 때 사용한다.

프롬프트:

```text
이번 작업을 sprint mode로 진행해줘.
너는 GPT-5.5 메인 운영자다.
GPT-5.4 서브 에이전트는 범위가 좁은 구현 또는 테스트 작업에만 사용해라.
이 기능을 승인된 범위에서 구현, 리뷰, QA, 배포 준비 상태까지 끌고 가라.

기능:
<기능 이름과 범위>
```

### 3. Triage Mode

버그, 안정성 문제, 원인 불명 이슈에 사용한다.

프롬프트:

```text
이번 작업을 triage mode로 진행해줘.
너는 GPT-5.5 메인 운영자처럼 행동해라.
먼저 문제를 식별하고, 그다음 조사, 좁은 수정, 더 큰 설계 변경 중 무엇이 필요한지 결정해라.
작업이 충분히 좁아졌을 때만 위임해라.

이슈:
<버그 또는 실패 설명>
```

### 4. Release Mode

이미 구현된 작업의 최종 출시 판단이 필요할 때 사용한다.

프롬프트:

```text
이번 작업을 release mode로 진행해줘.
너는 GPT-5.5 메인 운영자처럼 행동해라.
리뷰 준비 상태, QA 상태, 문서 상태, 배포 리스크를 검토하고 ship/no-ship 판단을 내려라.

범위:
<배포 대상 설명>
```

### 5. Maintenance Mode

신규 기능이 아니라 유지보수성, 안정성, 문서, 테스트 개선 작업일 때 사용한다.

프롬프트:

```text
이번 작업을 maintenance mode로 진행해줘.
너는 GPT-5.5 메인 운영자처럼 행동해라.
안정성, 정리, 테스트, 문서, 운영 단순화에 집중해라.
GPT-5.4 작업자는 범위가 좁은 실행 작업에만 사용해라.

유지보수 목표:
<작업 설명>
```

## 창업자 기본 SOP

이 항목은 가장 기본적인 운영 루틴으로 사용한다.

### 새로운 아이디어를 시작할 때

1. 문제를 평서문으로 설명한다
2. 메인 모델에게 planning mode를 요청한다
3. 범위를 압박 테스트한다
4. 가장 좁은 첫 버전을 승인한다

### 구현을 시작할 때

1. 기능 범위가 승인되었는지 확인한다
2. 메인 모델에게 sprint mode를 요청한다
3. 서브 에이전트 위임 여부는 메인 모델이 판단하게 둔다
4. 창업자는 중요한 트레이드오프에만 개입한다

### 버그가 발생했을 때

1. 메인 모델에게 triage mode를 요청한다
2. 이 이슈가 아래 중 무엇인지 구분한다
   - 단순 버그
   - 설계 문제
   - 운영 문제
3. 해결 경로를 승인한다
4. 재발 가능성이 있으면 회귀 테스트를 요구한다

### 배포 직전

1. 리뷰를 수행한다
2. QA를 수행한다
3. 문서를 확인한다
4. ship/no-ship 결정을 내린다
5. 배포를 막지 않는 후속 이슈는 별도로 기록한다

### 배포 직후

1. 초기 동작을 모니터링한다
2. 예상 밖의 문제를 기록한다
3. 새 표준으로 굳힐 패턴이 있으면 반영한다

## 확장 경로

이 문서는 창업자 중심으로 시작하지만, 아래 방향으로 확장할 수 있어야 한다.

### Stage A. 창업자 + AI

- 메인 운영자 1명
- 필요 시 서브 작업자 사용
- 가벼운 주간 계획

### Stage B. 소규모 인간 팀 + AI

추가할 것:

- 영역별 명시적 소유권
- 인간과 AI 사이의 인수인계 노트
- 팀 단위 주간 리뷰
- 더 명확한 배포 일정

### Stage C. 정식 회사 운영

추가할 것:

- 역할별 승인 게이트
- 팀 온보딩 문서
- 서비스 오너십
- 장애 대응 플레이북
- 정기 유지보수 예산

## 피해야 할 안티패턴

### 1. 서브 에이전트에게 제품 방향을 결정하게 하는 것

이렇게 하면 방향이 흔들리고 판단이 중복된다.

### 2. 범위가 안정되기 전에 위임하는 것

작업자는 결정을 실행해야지, 제품을 처음부터 다시 해석하면 안 된다.

### 3. 모든 작업을 멀티 에이전트 군집으로 돌리는 것

토큰을 많이 쓰고 통합 비용만 늘어난다.

### 4. 구현만 끝나면 바로 배포 가능한 것으로 보는 것

리뷰와 QA가 없는 구현은 회사 프로세스가 아니라 코드 출력에 가깝다.

### 5. 전략 작업과 좁은 버그 수정 작업을 한 스레드에 섞는 것

전략은 종합이 필요하고, 버그 수정은 속도와 선명함이 필요하다. 분리하는 것이 좋다.

`창업자는 목적지를 정하고, GPT-5.5는 경로를 정하고, GPT-5.4 작업자는 범위가 정해진 노동을 수행하며, gstack은 그 흐름에 운영 규율을 부여한다.`
